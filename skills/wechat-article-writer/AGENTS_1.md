# WeChat Article Writer - Multi-Agent System

## 概述

本系统使用多代理协作方式生成高质量微信公众号文章，通过专业分工和多层Review确保文章质量。

## 核心流程

```
用户输入主题
    ↓
【主代理】协调整体流程
    ↓
┌─────────────────────────────────────────┐
│  阶段1：专业分工创作                  │
├─────────────────────────────────────────┤
│  【子代理1】标题专家                   │
│  - 生成3个引人入目的标题               │
│  - 应用100+标题生成技巧                 │
│  - 输出：标题+技巧+理由                │
├─────────────────────────────────────────┤
│  【子代理2】结构专家                   │
│  - 设计文章段落架构                     │
│  - 确定文章类型和框架                  │
│  - 确保中心思想统一                    │
│  - 输出：结构大纲+中心思想              │
├─────────────────────────────────────────┤
│  【子代理3】内容专家                   │
│  - 根据结构和标题撰写内容               │
│  - 确保内容与结构、标题高度契合         │
│  - 输出：完整文章内容                  │
└─────────────────────────────────────────┘
    ↓
┌─────────────────────────────────────────┐
│  阶段2：三层Review机制                 │
├─────────────────────────────────────────┤
│  【Review 1】中心思想一致性检查         │
│  - 检查全文是否围绕中心思想展开         │
│  - 验证各段落是否服务于主题             │
│  - 确保没有偏离主题的内容              │
│  - 输出：一致性报告+修改建议            │
├─────────────────────────────────────────┤
│  【Review 2】设计与语言检查             │
│  - 检查设计元素（格式、排版、视觉）     │
│  - 检查错别字和语法错误               │
│  - 确保语义通顺，逻辑连贯              │
│  - 检查专业术语使用是否准确            │
│  - 输出：修改建议+优化方案             │
├─────────────────────────────────────────┤
│  【Review 3】去AI化与口语化优化        │
│  - 识别并去除AI写作痕迹               │
│  - 优化语言表达，使其更自然             │
│  - 增加口语化表达，提升亲和力          │
│  - 调整语气，符合目标受众习惯          │
│  - 确保文章有人情味，不生硬            │
│  - 输出：优化建议+最终版本             │
└─────────────────────────────────────────┘
    ↓
【子代理3】内容专家 - 根据Review结果优化内容
    ↓
┌─────────────────────────────────────────┐
│  阶段3：多版本生成                    │
├─────────────────────────────────────────┤
│  生成4个版本：                        │
│  1. 完整Markdown版（5000字）         │
│  2. 精简Markdown版（1000字）         │
│  3. 完整纯文本版（适合微信）          │
│  4. 精简纯文本版（适合微信）          │
└─────────────────────────────────────────┘
    ↓
完成，输出所有文件
```

## 子代理间数据传递格式

### 数据传递原则

**核心原则：**
- 每个子代理接收完整且独立的上下文
- 不依赖会话历史，所有必要信息显式传递
- 使用结构化格式（JSON或Markdown）确保可解析性
- 每个输出包含元数据（代理名称、时间戳、版本）
- 所有输出必须可独立理解和处理

### 数据格式规范

#### 通用元数据格式

每个子代理的输出都应包含以下元数据：

```markdown
---
Agent: {代理名称}
Timestamp: {ISO 8601时间戳}
Version: 1.0
Status: {DONE | DONE_WITH_CONCERNS | NEEDS_CONTEXT | BLOCKED}
---
```

#### 状态说明

- **DONE**: 任务完成，无问题
- **DONE_WITH_CONCERNS**: 任务完成，但有一些疑虑或观察
- **NEEDS_CONTEXT**: 需要更多信息才能继续
- **BLOCKED**: 无法完成任务，需要人工干预

**角色定位：**
- 专门负责生成引人入目的文章标题
- 应用100+种标题生成技巧
- 确保标题具有吸引力和传播性

**输入：**
- 文章主题
- 目标受众
- 文章风格

**输出：**
3个标题，每个标题包含：
- 标题内容（15-25字）
- 使用的技巧类型
- 生成理由

**标题生成技巧（100+种）：**

1. **反差型**：制造强烈对比
   - 示例："月薪3千 vs 月薪3万的程序员"
   - 适用：对比类、选择类文章

2. **悬疑型**：制造悬念
   - 示例："这个功能90%的人都不知道"

   - 示例："提升10倍效率的5个技巧"

   - 示例："还在为这个问题烦恼？"
   - 适用：解决方案类

5. **对比型**：比较选择
   - 示例："A vs B：谁更值得选择？"
   - 适用：产品对比、方案选择

6. **好奇型**：激发兴趣

   - 适用：知识科普、内幕揭秘

7. **紧迫型**：制造紧迫感

8. **独家型**：强调独特性

## Review机制

### Review 1：中心思想一致性检查

**检查要点：**
- 全文是否围绕中心思想展开
- 各段落是否服务于主题
- 是否有偏离主题的内容
- 中心思想是否贯穿始终

**输出：**
- 一致性评估（通过/需修改）
- 具体问题列表
- 修改建议

**处理流程：**
- 如果通过：进入Review 2
- 如果需修改：返回内容专家修改，然后重新Review

**循环机制：**
- 最大重试次数：3次
- 重试计数器：从0开始，每次修改后+1
- 重试策略：
  - 第1次修改：根据问题列表直接修改
  - 第2次修改：分析问题根源，针对性修改
  - 第3次修改：全面审查，系统性修改
- 超过最大重试次数：
  - 记录所有未解决的问题
  - 标记为"部分通过"
  - 继续进入Review 2
  - 在最终输出中明确标注未解决问题

**错误处理：**
- Review失败：
  - 记录错误详情（时间戳、错误类型、错误信息）
  - 返回BLOCKED状态
  - 提供错误原因和建议
- 内容专家修改失败：
  - 记录错误详情
  - 返回BLOCKED状态
  - 提供错误原因和建议
- 超时处理：
  - 单次Review超时：5分钟
  - 总Review超时：15分钟
  - 超时后记录并继续

7. **主代理**执行Review 2（设计与语言）

9. **主代理**执行Review 3（去AI化与口语化）
10. 如果Review 3未通过，返回步骤4修改，重新Review
11. **主代理**调用【子代理3：内容专家】生成4个版本的最终内容
12. **主代理**保存所有输出文件
13. **主代理**输出完成总结和下一步建议

## 全局错误处理

### 子代理失败处理

**标题专家失败：**
- 记录失败详情（时间戳、失败原因、输入参数）
- 返回BLOCKED状态
- 提供失败原因和建议
- 主代理决定是否使用默认标题或重新尝试

**结构专家失败：**
- 记录失败详情（时间戳、失败原因、输入参数）
- 返回BLOCKED状态
- 提供失败原因和建议
- 主代理决定是否使用默认结构或重新尝试

**内容专家失败：**
- 记录失败详情（时间戳、失败原因、输入参数）
- 返回BLOCKED状态
- 提供失败原因和建议
- 主代理决定是否使用简化内容或重新尝试

**Review Agent失败：**
- 记录失败详情（时间戳、失败原因、输入参数）
- 返回BLOCKED状态
- 提供失败原因和建议
- 主代理决定是否跳过该Review或重新尝试

### 文件保存失败处理

**保存失败处理流程：**
1. 检测到保存失败
2. 记录失败详情（文件路径、失败原因、时间戳）
3. 尝试备用保存位置
4. 如果备用位置也失败，记录到日志
5. 在最终输出中明确标注保存失败的文件

**备用保存位置：**
- 主位置：`wechat-articles/{topic-slug}/`
- 备用位置1：`wechat-articles/{topic-slug}-backup/`
- 备用位置2：`wechat-articles/fallback/{timestamp}/`

**文件保存验证：**
- 保存后验证文件是否存在
- 验证文件大小是否合理（>0字节）
- 验证文件内容是否完整
- 如果验证失败，记录并尝试重新保存
